## [4.9.2] - 2026-07-17

### Security
- **Real hostname-redaction gap found and fixed from a live production incident**: a user reported that an Azure OpenAI-generated report leaked two real hostnames (from a live Pacemaker/SAP HANA HA cluster) verbatim, despite redaction being enabled. Root cause: the bundle was a raw copy of `/var/log` with none of the dedicated system-info files (`sos_commands/`, `installed-rpms`, `basic-environment.txt`) that hostname detection depended on, so it was correctly detected as `kind="unknown"` - but `check_system_info()` only populates a hostname for `kind in ("sosreport", "supportconfig-shaped fallback")`, and `check_cluster_health()` (the other hostname source) only ever runs `if kind == "crm_report"`. For `kind="unknown"`, **neither ran**, so `collect_known_hostnames()` (`backend/ai/redaction.py`) had zero input to redact against - hostname redaction was a complete, silent no-op for this entire bundle, even though the real hostnames appeared in nearly every line of its own pacemaker/corosync logs.
  - Fixed with a new, content-based hostname detector (`sniff_syslog_hostnames()`, `backend/engine/analyzer_core.py`) that no longer depends on any bundle-kind-specific file existing at all: (1) a frequency-gated sniffer that recognizes the hostname field classic rsyslog/journald-style log lines embed immediately after the timestamp (requires 3+ identical repeats before trusting a candidate, since a real host's identifier repeats on nearly every line of a real log while a coincidental match doesn't); (2) a cluster-peer "sibling" detector that, once one node's hostname is confirmed, also trusts any other identifier sharing its exact prefix elsewhere in the bundle (e.g. confirming `ue2op1dbsp01` also trusts `ue2op1dbsp02`) - since a peer's name legitimately appears only in free-text corosync/pacemaker membership and fencing messages, never in the local host's own syslog hostname field, and HA-cluster nodes overwhelmingly follow a shared-prefix-plus-incrementing-node-number naming convention.
  - Runs unconditionally (any bundle `kind`) as a bounded, capped sample (max 30 candidate files, 100KB/300 lines each - a fixed, small cost even on a 90,000+-file bundle) - also valuable as a supplementary source for sosreport/supportconfig/crm_report bundles, where it can additionally catch a cluster peer's hostname that the existing per-kind sources don't track.
  - Verified end-to-end with 21 automated checks, including a full reconstruction of the exact reported scenario (a raw `/var/log`-shaped bundle with a compressed `pacemaker.log-*.xz`, two real-shaped HA cluster hostnames, `kind="unknown"`) proving both hostnames are now correctly redacted before the digest would reach any non-local AI provider - plus regression checks confirming the existing sosreport/crm_report hostname sources are unaffected (the new source is additive) and that a 2-field syslog line with no hostname field (some minimal configs omit it) is correctly not mistaken for one.
  - **This fix does not undo an exposure that already happened** - any report already generated from a `kind="unknown"` bundle before this version had its hostnames sent to whichever provider generated that report; only future analyses are protected. See SECURITY.md's new "A real incident: hostnames leaked despite redaction" section for the full, honest writeup.

## [4.9.1] - 2026-07-17

### Security
- **PII-redaction coverage audit and fixes, prompted directly by a user question ("is redaction actually working, and could anything have silently weakened it?").** Investigation confirmed the WIRING was intact - no recent change had disconnected redaction, every provider added since (Mistral AI, DeepSeek, GitHub Models) was already correctly marked non-local so redaction applies to them too, and the UI checkbox is not persisted/rememberable in a way that could leave it silently unchecked. However, three real gaps were found and fixed in the redaction *logic* itself (`backend/ai/redaction.py`):
  - **IPv6 addresses were not redacted at all** - only IPv4 was ever handled. Fixed with a candidate-regex + stdlib `ipaddress.ip_address()` validation approach (correctly handles compressed `::` forms and IPv4-mapped addresses like `::ffff:192.168.1.1` without a hand-rolled, error-prone monster regex; confirmed to correctly ignore lookalikes such as ordinary `HH:MM:SS` timestamps and MAC addresses).
  - **A known hostname was not redacted when it appeared fully-qualified** (e.g. a known short hostname `ue2op1dbsp01` inside `ue2op1dbsp01.corp.contoso.com`) - the previous word-boundary check treated `.` as "still part of the same identifier", which is backwards for this purpose; the boundary now correctly allows a `.` on either side while still preventing false matches against a longer, unrelated alphanumeric identifier.
  - **Email addresses were never redacted** - added as a new category (`EMAIL-1`, `EMAIL-2`, ...).
  - The Results-page redaction callout (`frontend/app.js`) only ever grouped legend entries into "hostnames" or "IPs" by token prefix - the new `IPV6-`/`EMAIL-` tokens would have been silently dropped from the on-screen summary/mapping (though still correctly redacted from the actual outbound request) had this not been caught and fixed in the same pass.
  - Documented (rather than silently left as an unstated gap) the one residual-risk category that remains: crm_report's own "Built-in Analysis (analysis.txt, verbatim)" digest section embeds up to 4000 characters of freeform third-party text that can mention a hostname this analysis never independently identified elsewhere (e.g. a fencing device or NFS peer, not a cluster node) - IPv4/IPv6/email addresses inside that same block are still caught (shape-based, not allow-list based), but an unknown hostname mentioned only there is not. See the "Limitations" section of `backend/ai/redaction.py`'s module docstring and the updated SECURITY.md "Redaction" section for the full, honest picture.
  - Verified with 51 automated checks: unchanged pre-existing IPv4/hostname behavior (no regression), every new IPv6 form (link-local, compressed, full, loopback, IPv4-mapped), IPv6 false-positive resistance against timestamps/MAC addresses, every legitimate RHEL/SUSE/ClusterLabs reference doc link used elsewhere in the digest confirmed to never be mistakenly redacted, the FQDN-boundary fix (plus a check that it does *not* over-match a longer identifier merely containing a known hostname as a substring), new email-address redaction, an end-to-end scenario mirroring the crm_report verbatim-block residual-risk case above, and every registered AI provider confirmed correctly marked local/non-local.

### Added
- **Dynamic, chunk-based work distribution for parallel scanning** - fixes a real scheduling inefficiency directly diagnosed from a user's live production `top` output: with v4.9.0's exactly-one-static-batch-per-worker model, a worker whose batch happened to contain disproportionately expensive files (LPT partitioning balances by total *bytes*, but actual scan cost isn't perfectly proportional to bytes alone - regex-match density varies by file content) could run long after every other worker had already finished and exited, leaving those CPU cores completely idle for the remainder of the run even though the machine had spare capacity. Now, `worker_count * _CHUNKS_PER_WORKER` smaller chunks (still LPT-balanced by bytes via the existing `_partition_items_by_size()` helper) are submitted to `ProcessPoolExecutor(max_workers=worker_count)` all at once, instead of exactly `worker_count` large ones - `concurrent.futures`' own internal task queue then dynamically hands a newly-free worker the next pending chunk immediately, achieving genuine load-balancing using only existing stdlib behavior, no custom work-stealing scheduler needed. A `_MIN_ITEMS_PER_CHUNK` floor prevents excessive fragmentation (and its per-chunk IPC overhead) when a very high explicit worker count is requested against a comparatively modest bundle. Progress reporting was correspondingly rewritten from "worker N/total finished" to "X/N chunks done, Y active now" phrasing, since a chunk no longer maps 1:1 to a single worker process for the run's whole duration.
- **Raised the auto-detection worker ceiling from 8 to 16**, and added a separate, more generous `_ABSOLUTE_MAX_WORKERS = 128` ceiling that applies only to an *explicit* worker-count request (not auto-detection) - deliberately not capped at this machine's CPU count for an explicit request, since oversubscription can genuinely help an I/O-wait-dominated workload (e.g. many small files under real-time antivirus/EDR scanning overhead) even though it won't help a CPU-bound one.
- **"Parallel scanning workers" dropdown now shows this machine's own core count for context** (e.g. "16 (2.0x this machine's 8 cores — oversubscribed)") via a new `GET /api/system-info` endpoint, so choosing a higher worker count is an informed choice rather than a guess. Expanded the dropdown itself to include 16/32/64/128 in addition to the existing Auto/1/2/4/8.
- **Mid-file progress reporting** - a single genuinely huge file (real customer bundles have had individual rotated pacemaker/perfdata logs in the 78-120MB range) could previously cause a multi-minute silent gap in the activity log even with per-file reporting one level up, since nothing reported progress *within* one file's scan. `scan_file()` now accepts a `progress_cb` checked every 5,000 lines; wired into both the sequential path and (via a `multiprocessing.Manager().Queue()`, polled every 0.5s alongside batch-completion checks) the parallel path, so a long-running chunk's current file/line position surfaces live instead of only being visible once an entire chunk finishes.

Verified with 19 automated checks: a scheduling-math simulation directly proving the new dynamic-chunking approach produces a measurably lower makespan than the old static-partitioning approach on a deliberately skewed workload shaped like the real bundle that motivated this fix; chunk-count sizing-formula edge cases (including the `_MIN_ITEMS_PER_CHUNK` anti-fragmentation guard against a high explicit worker count on a modest bundle); a full end-to-end `run_analysis()` test confirming sequential and forced-parallel (actually exercising the new chunked `ProcessPoolExecutor` path) produce identical finding signatures and file/compressed-file counts; and the exact background `threading.Thread(daemon=True)` invocation pattern `backend/app.py`'s real job runner uses.

## [4.9.0] - 2026-07-17

### Added
- **Parallel file scanning, auto-tuned to the machine's CPU/memory** - the line-by-line scan (the CPU-bound part of analysis: regex matching against every line of every file) now runs across multiple worker **processes** instead of one, when the bundle is big enough for that to actually help. Directly addresses feedback that a large real bundle (tens of thousands of files, several 100MB+ logs) took a long time to analyze - regex matching is CPU-bound, not I/O-bound, so Python **threads** wouldn't have helped (the GIL means only one thread executes Python bytecode at a time regardless of thread count - directly confirmed earlier by watching a single scan peg one CPU core at ~100% for the whole run); genuine parallelism for CPU-bound work in Python needs separate processes, each with its own GIL.
  - `determine_worker_count()` (`backend/engine/analyzer_core.py`) auto-detects a sensible worker count from `os.cpu_count()`, available memory (OS-native detection - `ctypes`/`GlobalMemoryStatusEx` on Windows, `/proc/meminfo` on Linux - no new dependency, matching this project's stdlib-only philosophy everywhere else), and bundle size. Small bundles (under ~300 files or ~20MB of scannable content) stay fully single-process automatically, since process-pool startup overhead wouldn't pay off - this only engages when it's actually worth it.
  - `_partition_items_by_size()` uses a greedy longest-processing-time-first (LPT) partitioning so work is balanced by total **bytes**, not just file count - directly matters for real bundles where a handful of individually-huge logs dominate scan time among thousands of small ones (exactly the shape of the real bundle that motivated this feature); naive equal-count chunking would leave one worker stuck with all the huge files while others sit idle.
  - New optional `workers` parameter on `run_analysis()` (and `--workers` on the CLI, and a "Parallel scanning workers" dropdown in Step 1 → Advanced options, defaulting to "Auto"): omit/leave at Auto to auto-detect (recommended), or pass `1` to force the original single-process sequential scan (useful for timing comparisons or ruling out a parallelism-related issue), or any other number to pin an explicit worker count.
  - The original sequential code path is completely untouched as the `workers<=1` branch - the multi-worker path is strictly additive, not a rewrite of the already-proven-correct single-threaded scan. If the process pool itself fails to start for any reason (a sandboxed environment forbidding subprocess creation, etc.), analysis automatically falls back to the sequential path with a clear log line, rather than aborting.
  - One disclosed, minor behavioral nuance versus strictly sequential scanning: when a finding occurs across files handled by different workers, the specific handful of example file:line references shown for it (capped at a few per finding, same as always) can differ slightly from "always the first N encountered in file order" - the total occurrence **count** is always exact and complete either way; only the particular sample examples shown might vary, since those have always been "some representative samples," not an exhaustive or canonically-ordered list.

Verified with 61 automated checks across 4 test suites: unit tests for the worker-count/partitioning/memory-detection helpers; a full end-to-end test on a realistic synthetic bundle confirming sequential vs. forced-parallel vs. auto-detected runs all produce **exactly** equivalent total finding counts/categories/line counts (and measured a genuine 1.4-4.3x speedup depending on file-size distribution, not just "compiles and doesn't crash"); a dedicated test confirming this works correctly when invoked from within a background `threading.Thread(daemon=True)` - the exact way `backend/app.py`'s real job runner invokes it, and the specific Windows-multiprocessing interaction that mattered most to get right; and a combined test confirming v4.8.0's compressed-file decompression (gzip/bz2/xz, including double-compressed `.xz.gz`) works correctly *inside* a spawned worker process, with finding provenance still correctly pointing at the real compressed file path.

## [4.8.0] - 2026-07-17

### Fixed
- **Individually-compressed rotated log files (`.gz`/`.bz2`/`.xz`, including stacked/double compression) were silently skipped as "binary" and never scanned** - reported from a real customer bundle with rotated `messages-*.gz` logs going back to 2018, plus a newer, unusual double-compressed pattern (`messages-20260406.xz.gz` - an xz-compressed log gzipped again on top, most likely by a separate backup/transfer step). Before this release, `.gz`/`.bz2`/`.xz` were all in the binary-skip extension list, so every one of these files was inventoried but silently never scanned for failure signatures - a real, silent evidence gap, since a rotated log from just before an incident is often exactly the evidence a root-cause investigation needs.

### Added
- **Transparent decompression for gzip/bz2/xz-compressed files** (`backend/engine/analyzer_core.py`, stdlib-only - no new dependency): detection is by **content (magic bytes), not filename extension** - `sniff_compression_kind()` peeks the first few bytes and recognizes gzip/bz2/xz regardless of what the file is named, and `open_decompressed_binary()`/`open_decompressed_text()` peel compression layers one at a time, re-checking after each peel, so both a normal single-compressed file and a genuinely stacked one (like the real `.xz.gz` case above) are handled correctly without needing to know in advance which one it is - up to 3 layers, generous for real-world cases.
  - `classify_file_readability()` replaces the old binary/text-only decision in the main scan loop with a three-way one (`"text"` / `"compressed"` / `"skip"`) - still exactly **one** file open + read per file (no added I/O pass over the common non-compressed case, verified via a direct before/after timing comparison over 2,000 synthetic files - the new function was not measurably slower).
  - A compressed file's *decompressed* content is also verified to look like text (not e.g. a gzipped binary state dump some backup tools produce) before it's scanned - a compressed non-text blob is still correctly skipped, not fed through the line scanner as garbage.
  - `get_text()` - the helper nearly every structured fact-check (`check_security_mac`, `check_package_drift`, etc.) uses to read a specific known file - is now decompression-aware too, so every analyzer that reads files via `find_inventory()` + `get_text()` gets compressed-file support for free, with zero changes needed in any individual check function.
  - A 2 GB decompressed-size safety cap (`scan_file()`) guards specifically against a corrupted or maliciously-crafted archive expanding without bound (a "zip bomb" risk that's unique to compressed input - a plain-text file's cost is already bounded by its own on-disk size) - never expected to trigger on legitimate rotated logs, which compress at best a few dozen:1.
  - The digest's "Files content-scanned" line and the live progress log now both mention how many files were transparently decompressed, so this is visibly confirmable on any real run, not just a silent behind-the-scenes change.
- Deliberately out of scope for this release (documented in code comments, not silently ignored): multi-member archive formats (`.zip`, `.tar`, `.tar.gz`) are unchanged - those are container formats that can hold many files, a materially different (and riskier - path traversal, "which member(s) do we scan") feature than the real-world need this addresses: individual rotated-log-style files compressed with gzip/bz2/xz. A full sosreport/supportconfig/crm_report bundle that is itself a `.zip`/`.tar.*` continues to be handled exactly as before, at extraction time.

Verified with: 26 direct unit checks (gzip/bz2/xz single-layer, the exact double-compressed `.xz.gz` customer scenario, a corrupted/truncated archive, a compressed genuinely-binary blob correctly still skipped, `get_text()` decompression, zero regression on plain uncompressed files); a full `run_analysis()` end-to-end test recreating the customer's reported bundle shape (rotated `.gz`/`.bz2`/`.xz`/`.xz.gz` messages logs) confirming findings are correctly extracted from every compressed variant with provenance still pointing at the real compressed file path; a 13-check regression pass confirming all pre-existing analyzer categories (SAR, crash, boot, security, packages, cascade, containers) and the OS-knowledge module are unaffected on both sosreport and supportconfig fixtures; and a direct timing comparison confirming no meaningful performance regression on the common non-compressed path.

### Added
- **Dedicated Results tab per analyzer** (💥 Crash, 🥾 Boot, 🛡️ Security, 📦 Packages, 🔗 Cascade, 🐳 Containers, 🌐 Network, plus the new 🐧 OS & Config below) - each of the fact-producing checks now gets its own tab showing its actual structured data, not just prose buried in the Digest, or the explicit message **"No relevant data found from the sosreport or supportconfig."** (with a short reason, e.g. Network noting that packet capture isn't part of a standard bundle) when that category genuinely has nothing to show. Built after independently re-verifying, category by category, that every check does correctly extract real data from realistic sosreport AND supportconfig fixtures (43 automated checks, both bundle types, plus an empty-bundle sanity check) - the "is this actually finding anything by default?" concern traced to one genuine bug (see Fixed, `try_parse_full_date`), not a broader problem with the checks themselves.
- **Linux OS/version knowledge base** (`backend/engine/os_knowledge.py`, new): detects OS family/major/minor version from whatever release-identification file the bundle contains (`/etc/os-release`, legacy `/etc/redhat-release`, legacy `/etc/SuSE-release`, or the supportconfig equivalents), then attaches:

- **`trust-cert.ps1`/`trust-cert.sh`/`trust-cert.bat`**: one-command fix for the "connection isn't private" warning on `--https`'s self-signed certificate - imports it into the OS's trusted-root store so the *same* browser that showed the warning loads the page with zero warning afterward. (Doesn't help a *different* machine/browser connecting to the same server - that's inherent to self-signed certificates, not specific to this tool; use your own CA-issued certificate via `--ssl-certfile`/`--ssl-keyfile` for that case.)

### Fixed
- **`check_package_drift()` silently found zero results on any real-world timezone other than UTC/GMT**: `try_parse_full_date()` used Python's `%Z` `strptime` directive, which only reliably parses `"UTC"`/`"GMT"` - `"IST"`, `"EST"`, and effectively every other real-world timezone abbreviation from `rpm -qa --last` output silently failed to parse. This was likely a real contributor to package-drift analysis appearing to "not find anything" on real customer bundles. Fixed by stripping the timezone-abbreviation token before parsing instead of relying on `%Z` to consume it. Verified with 6 direct test cases plus regression coverage in the wider 43-check analyzer suite.

## [4.6.0] - 2026-07-16

### Added
- **Stop scripts** (`stop.sh`/`stop.bat`/`stop.ps1`) - the counterpart to `run.sh`/`run.bat`/`run.ps1`, for stopping a server you can't just `Ctrl+C` (started detached/in the background, or in a window/session you're no longer attached to). Each one: (1) if the server responds, calls its own `POST /api/ollama/stop` first so any Ollama instance it's managing gets stopped too - reusing the exact same "only ever stop what we started" safeguard as the in-app Stop button; (2) finds whatever process is actually listening on the target port and verifies its command line looks like LDI Copilot (`app.py`/`uvicorn`) before stopping it, printing a warning and leaving it alone otherwise unless `--force`/`-Force` is passed; (3) optionally, with `--kill-ollama`/`-KillOllama`, force-stops every Ollama process on the machine regardless of who started it (off by default, mirroring the in-app Stop button's own conservative default).

### Fixed
- **`stop.sh` silently failed to actually stop anything when run via Git Bash on Windows** - caught during testing, before release. Three separate, compounding issues specific to Git Bash/MSYS2 (none of which affect real Linux/macOS/WSL, where the script worked correctly from the start):
  1. None of `ss`/`lsof`/`fuser` exist in a default Git Bash install, so the port-to-PID lookup silently found nothing at all - fixed with a `netstat.exe`-based fallback (Windows' own netstat is reachable from Git Bash via PATH).
  2. Git Bash's `ps` is a from-scratch reimplementation with no view into a real Windows process's actual command line, so the "does this look like our server" safety check always saw a blank command line and refused to proceed - fixed with a PowerShell/WMI (`Get-CimInstance Win32_Process`) fallback for reading it.
  3. The most subtle one: MSYS bash's `kill`/`kill -0` builtins silently no-op on a genuine Windows PID that wasn't spawned from within that same MSYS process tree, rather than erroring - so the script went on to print "stopped" even though nothing had actually happened. Fixed by detecting that case (no corresponding `/proc/<pid>` entry) and falling back to `taskkill.exe`/`tasklist.exe` instead, which work correctly regardless of which subsystem started the target process - and setting `MSYS_NO_PATHCONV=1`, since Git Bash's automatic POSIX-to-Windows path conversion otherwise mangles bare `/PID`/`/FI` switches (e.g. rewriting `/PID` into a literal Windows path) before `taskkill.exe`/`tasklist.exe` ever see them.

  Caught by genuinely running the script via Git Bash and checking the real process/port state afterward at each step, rather than trusting the script's own "Done - stopped" message - a good reminder that a script reporting success and a script actually having succeeded are two different claims worth verifying independently, especially when shelling out across a subsystem boundary like MSYS2/Git-Bash-on-Windows.

Verified via live end-to-end runs of all three scripts against a real running server on Windows: PowerShell (`stop.ps1`) directly, Command Prompt (`stop.bat`, via a genuine interactive `cmd.exe` session), and bash (`stop.sh`, via Git Bash) - each covering the normal stop path, the "nothing running" idempotent case, the safety check correctly refusing to stop an unrelated process occupying the same port, and `--force`/`-Force` correctly overriding that refusal when passed.

## [4.5.0] - 2026-07-16

The check is best-effort and non-blocking: missing credentials, invalid keys, or network failures simply leave every option selectable with an explanatory status message, never a hard error.
- `backend/ai/providers.py` gained `list_models()` / `list_models_openai()` / `list_models_anthropic()` / `list_models_ollama()` and a `KNOWN_MODELS` registry; `backend/app.py` gained `POST /api/models` (always returns HTTP 200 with `{available, error}`, never raises, for a failed live check).

Uses the standard OAuth2 client-credentials flow against `https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token` (scope `https://cognitiveservices.azure.com/.default`), then calls Azure OpenAI with an `Authorization: Bearer` header. A fresh token is requested per synthesis call; nothing is cached or written to disk. See the README for full app-registration/RBAC setup steps.

- A small Microsoft logo mark (inline SVG, no external asset) next to the app name in the header.

### Fixed
- Fixed a design flaw in the previous panel-relocation mechanism where starting a second analysis from "Provide Bundle" after visiting "Results" could reference stale/detached DOM nodes; the new architecture (panel never moves, tabs just show/hide) eliminates that class of bug entirely.

## [1.1.0] - 2026-07-13

### Added
- **Focused analysis**: a new "What are you investigating?" field, front and center in Step 1, lets you describe the specific issue you want root-caused (e.g. "find root cause of NC and IP cluster resource restart issue") instead of always getting a generic exhaustive report.
  - The mechanical engine (`backend/engine`) extracts keywords from the focus text and tags every finding/timeline event with `focus_match`. The digest gains a dedicated `## 🎯 Focused Findings` section ahead of the full category breakdown, and matches are marked with 🎯 throughout.
  - Keyword matching treats `_`/`-` as separators (not just whitespace), so it correctly matches short identifiers like `ip`/`nc` embedded in real-world Pacemaker resource names such as `rsc_ip_cluster` or `rsc_nc_share`.
  - The AI synthesis prompt (`backend/ai/prompts.py`) is restructured around the stated focus when one is given: the executive summary and root-cause section answer the focus directly, evidence outside the focus is demoted to a short closing "Other Observations" section, and the model is explicitly told to reason beyond the mechanical keyword matches (e.g. connecting a flapping NIC to an IP-resource restart even though "NIC" and "IP" share no literal keyword).
- **AI model selection moved to Step 1**: the provider/model/API-key configuration panel that used to live at the end (in the AI Root Cause Report tab) now lives in Step 1, alongside the new focus field. Fill it in before running analysis and a full AI-reasoned report is generated **automatically** as soon as the mechanical scan finishes - no extra click needed. Leave it blank to just get the mechanical evidence digest and configure AI later; the same panel relocates into the AI tab afterward so it can still be filled in or tweaked to regenerate without re-running the scan.
- Results now open on the **AI Root Cause Report** tab by default (previously Digest), since that's normally the first thing you want to read.
- A "🎯 Focused on: ..." callout summarizing the stated focus and match count appears above the results tabs.
- The Findings tab gained a "Show only findings matching my focus 🎯" toggle (checked by default when a focus was used) to cut through unrelated noise (e.g. an unrelated STONITH/fencing resource's routine warnings) when investigating one specific resource or subsystem.
- `run_analysis()` / the CLI gained a `focus` parameter / `--focus` flag for the same capability outside the browser UI.
- New synthetic test fixture `samples/fake_crm_report_multi`: a 2-node cluster with two independent problems (an Azure STONITH resource with unrelated API rate-limit noise, and a genuine NIC-flap-induced restart of `rsc_ip_cluster`/`rsc_nc_share`) - used to validate that focus steering correctly promotes the relevant evidence and demotes the unrelated one.

### Changed
- `backend/ai/prompts.py`: `build_messages()` now accepts a `focus_text` parameter; `/api/jobs/{id}/synthesize` accepts an optional `focus_text` override and otherwise reuses the focus text supplied at analysis time.
- `POST /api/analyze` accepts a new optional `focus` form field, stored on the job and threaded through to the mechanical scan.

## [1.0.0] - 2026-07-11

### Added
- Initial release: local browser-based AI agent for root-cause analysis of sosreport, supportconfig, and crm_report/hb_report diagnostic bundles.
- `backend/engine`: mechanical scanning engine (extraction, format detection, pattern matching, structured fact-checks, cross-file timeline, evidence digest) extended from the standalone `sosreport-rca` CLI tool with full crm_report support (per-node categorization, cluster status facts, analysis.txt cross-check) and refactored into an importable `run_analysis()` library API.
- `backend/ai`: pluggable AI provider clients (OpenAI, Anthropic Claude, Azure OpenAI, local Ollama) behind one streaming chat interface, plus the root-cause-synthesis prompt template.
- `backend/app.py`: local FastAPI server - upload/analyze jobs with background execution and progress polling, results API (digest/findings/facts/inventory/timeline), and an AI synthesis endpoint streaming the model's report via Server-Sent Events.
- Dependency-free vanilla HTML/CSS/JS frontend: drag-and-drop upload, time-window scoping controls, results dashboard, AI provider/model picker with optional per-device credential storage, and a small built-in Markdown renderer (zero CDN dependencies).
- `samples/`: synthetic `fake_sosreport`, `fake_supportconfig`, and `fake_crm_report` fixtures.
- `run.ps1`: one-command local launcher (venv + deps + server + browser).
